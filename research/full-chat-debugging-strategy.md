## Recommendations — skills, not MCP

Same feedback loops, no always-on connections. Package each capability as a skill: a short `SKILL.md` (when + how) plus a hardened script the agent runs on demand. Leaner than MCP — nothing loaded into context until the task needs it, no server processes, no 137-tool surface bloating your window. Pull-based (agent shells out, reads stdout) instead of push-based, which for debugging is exactly right: you query logs/traces/browser/DB only when chasing a specific bug.

**Skill layout:**

```
.agent/skills/
  query-observability/   SKILL.md + o2_search.sh    # wraps POST /api/{org}/_search → JSON
  inspect-browser/       SKILL.md + capture.mjs      # Playwright: console + network + screenshot
  query-db/              SKILL.md + psql_ro.sh       # read-only role, refuses non-SELECT
  profile-node/          SKILL.md + profile.sh       # clinic doctor/flame/heapprofiler
```

**`query-observability/SKILL.md`** (the whole interface the agent needs):

```markdown
---
name: query-observability
description: >
  Query OpenObserve logs and traces when debugging a production/staging error,
  slow request, or failed deploy. Use when a trace_id, error message, or service
  name is known and you need runtime evidence.
---
# Querying OpenObserve
Run `./o2_search.sh <logs|traces> "<SQL>" <minutes_back>`. Output is JSON.

## Recipes
- Recent errors:     ./o2_search.sh logs "SELECT _timestamp, service_name, message, trace_id FROM default WHERE level='error' ORDER BY _timestamp DESC" 15
- Follow a trace:    ./o2_search.sh traces "SELECT operation_name, duration, status_code FROM default WHERE trace_id='<id>' ORDER BY start_time" 60
- Slowest endpoints: ./o2_search.sh traces "SELECT operation_name, approx_percentile_cont(duration,0.99) p99 FROM default GROUP BY 1 ORDER BY 2 DESC" 60
- Error burst:       ./o2_search.sh logs "SELECT histogram(_timestamp) t, count(*) c FROM default WHERE level='error' GROUP BY t" 30

## Rules
- Smallest time window that could contain the evidence.
- Times auto-converted to microseconds by script. Don't compute them.
- SELECT only.
```

Script underneath = the ~20-line `curl` wrapper around `/api/{org}/_search` (computes µs timestamps, injects Basic Auth from env, prints JSON). No Enterprise license, no MCP server, works on your OSS instance.

## Summary

The fastest debugging is what an agent does for you in a closed loop, and that loop is only as good as the signal fed back after each change. Highest leverage: deterministic verification (a Stop hook running `tsc --noEmit` + focused Vitest that blocks until clean), a dense command-first AGENTS.md/CLAUDE.md so the agent knows your exact commands, and `@opentelemetry/instrumentation-pino` injecting `trace_id`/`span_id` into every Pino record so one ID stitches a stack trace → span → slow SQL query in OpenObserve. Instead of always-on MCP servers, package each runtime signal — observability queries, browser console/network capture, read-only DB, Node profiling — as an on-demand skill: a short `SKILL.md` plus a hardened script the agent shells out to only when a bug calls for it. Lean context, no Enterprise license (the observability skill wraps the open `/api/{org}/_search` API directly), and "why is this broken in staging" collapses into one skill invocation that assembles the trace, surrounding logs, and relevant queries in seconds.

Underneath the agent layer, classic technique decides how fast you find the bug. Interactive debuggers and conditional breakpoints/logpoints over `console.log`; attach to remote/containerized Node via `--inspect` over an SSH tunnel or `kubectl port-forward` (never expose 9229); `clinic doctor → flame/bubbleprof/heapprofiler` for perf and leaks; `git bisect run` to binary-search regressions. Shift left with Zod contracts at IO boundaries and `--enable-source-maps` so stacks point at `.ts`. Framework traps: debug hydration only in production builds by diffing server vs client HTML and bisecting the tree; log Prisma queries as spans to surface N+1s; for Python, `pytest -x --pdb` with `--lf` and rich tracebacks. The OTel pipeline fails more often than your code — when spans vanish, verify SDK → Collector (`debug` exporter, `otelcol_receiver_accepted_spans`) → backend in that order, and remember 4317 is gRPC, 4318 is HTTP.
